## 一、作業需求（翻譯自 README）

這是一份後端工程師的技術作業：用 **Spring Boot** 做一個 RESTful 通知服務，整合 **MySQL、Redis、RocketMQ**。

### 目標
設計並實作一個通知 API 系統，讓使用者可以發送 **email** 或 **sms** 通知。系統必須：
- 把通知**存進 MySQL**
- 把訊息**推送到 RocketMQ**
- 用 **Redis 快取**最近的通知並可讀取

### 技術要求（必須使用）
- Java 21+
- Spring Boot
- MySQL（持久化）
- Redis（快取）
- RocketMQ（事件訊息）

### 要實作的 5 個功能

| # | 功能 | Endpoint | 行為 | 狀態碼 |
|---|---|---|---|---|
| 1 | 建立通知 | `POST /notifications` | 存進 MySQL → 送到 RocketMQ topic `notification-topic` → 加入 Redis 最近 10 筆快取 | 201 |
| 2 | 依 ID 查詢 | `GET /notifications/{id}` | 先讀 Redis；沒有就讀 MySQL 並回寫 Redis | 200 / 404 |
| 3 | 查最近通知 | `GET /notifications/recent` | 回傳最近 10 筆 | 200 |
| 4 | 更新通知 | `PUT /notifications/{id}` | 更新 MySQL 的 subject/content；Redis 同步更新或失效；不存在回 404 | 200 / 404 |
| 5 | 刪除通知 | `DELETE /notifications/{id}` | 從 MySQL 刪除；移除 Redis 快取；不存在回 404 | 204 / 404 |

**建立通知的請求格式**
```json
{
  "type": "email",              // 或 "sms"
  "recipient": "user@example.com",
  "subject": "Welcome!",
  "content": "Thanks for signing up!"
}
```

**更新通知的請求格式**（只能改這兩欄）
```json
{
  "subject": "Updated subject line",
  "content": "Updated content of the notification"
}
```

### 加分項（選做）
- 用 Spring Cache 抽象或 RedisTemplate 封裝
- 考慮 race condition（並發競態）
- 適當的錯誤處理與有意義的狀態碼
- 自訂 DTO 與 RocketMQ 訊息格式
- 模組化的程式碼結構（controller / service / repository / config）
- 測試覆蓋率盡量高

### 環境
用 `docker-compose up -d` 啟動：MySQL(3306)、Redis(6379)、RocketMQ Namesrv(9876)、Broker(10911)、Console(8088)。
MySQL 帳密：taskuser / taskpass，資料庫 taskdb。預期完成時間約 3 小時。

## 二、系統架構

採**分層架構（Layered）+ Cache-Aside（旁路快取）**：

```
HTTP ─▶ Controller ─▶ Service ─┬─▶ Repository(JdbcTemplate) ─▶ MySQL（真相來源）
                               ├─▶ CacheService(RedisTemplate) ─▶ Redis（加速器）
                               └─▶ Producer(RocketMQ 原生 client) ─▶ RocketMQ
```

### 各層職責

| 層 | 類別 | 職責 |
|---|---|---|
| Controller | `NotificationController` | 5 個 REST 端點、HTTP 狀態碼 |
| DTO | `CreateNotificationRequest` / `UpdateNotificationRequest` / `NotificationResponse` / `NotificationMessage` | 請求 / 回應 / MQ 訊息格式 |
| Service | `NotificationService` | 編排 DB + 快取 + MQ，核心商業邏輯 |
| Repository | `NotificationRepository` | JdbcTemplate 存取 MySQL（參數化 SQL，防注入） |
| Cache | `NotificationCacheService` | by-id 快取 + 最近 10 筆，含降級處理 |
| Messaging | `NotificationProducer` + `RocketMQConfig` | 建立 Producer、送 `notification-topic` |
| Config | `RedisConfig` | RedisTemplate JSON 序列化（支援 Instant 時間） |
| Exception | `GlobalExceptionHandler` + `NotFoundException` | 統一錯誤處理（404 / 400 / 500） |
| Domain | `Notification` + `NotificationType`(列舉) | 領域模型 |

### 資料表 `notification`

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | BIGINT 自增 | 主鍵 |
| type | VARCHAR(16) | email 或 sms |
| recipient | VARCHAR(255) | 收件人（信箱或手機） |
| subject | VARCHAR(255) | 主旨 |
| content | TEXT | 內容 |
| created_at | DATETIME | 建立時間 |
| updated_at | DATETIME | 更新時間（PUT 時更新） |

### 快取策略
- **by-id**：key 為 `notification:{id}`，存 JSON，有 TTL（預設 3600 秒）。
- **最近 10 筆**：key 為 `notifications:recent`，用 Redis List（LPUSH 推到最前、LTRIM 裁到 10 筆）。
- MySQL 是唯一真相來源；Redis 只是加速器。

## 三、需求規格（S-SRS）

### 功能性需求
| 編號 | 需求 |
|---|---|
| SRS-F-01 | 建立通知 → 寫 MySQL → 送 RocketMQ → 加入 Redis 最近清單，回 201 |
| SRS-F-02 | 依 ID 查詢 → 先 Redis，未命中讀 MySQL 並回寫，200/404 |
| SRS-F-03 | 查最近通知 → 回最近 10 筆（優先 Redis，未命中由 DB 重建） |
| SRS-F-04 | 更新通知 → 改 subject/content；Redis 更新並讓 recent 失效；不存在 404 |
| SRS-F-05 | 刪除通知 → 刪 MySQL + 移除 Redis；不存在 404，成功 204 |

### 介面 / 資料需求
| 編號 | 需求 |
|---|---|
| SRS-I-01 | 建立請求含 `{type, recipient, subject, content}`，type 限 email/sms |
| SRS-I-02 | 更新請求僅 `{subject, content}` 可改 |
| SRS-I-03 | recent 回應含 id, type, recipient, subject, createdAt（ISO-8601 UTC，結尾帶 Z） |
| SRS-I-04 | 自訂 RocketMQ 訊息 DTO，topic 固定 `notification-topic` |
| SRS-D-01 | 持久化於 MySQL taskdb，schema 由 init.sql 建立 |
| SRS-D-02 | Redis 維護「最近 10 筆」與「by-id」兩種快取 |

### 非功能性需求
| 編號 | 需求 |
|---|---|
| SRS-N-01 | 技術棧：Java 21、Spring Boot 3.5.3、MySQL 8、Redis 7、RocketMQ 5.1.4 |
| SRS-N-02 | docker-compose 一鍵啟動 |
| SRS-N-03 | 模組化分層結構 |
| SRS-N-04 | 統一錯誤處理與語意化狀態碼 |

## 四、資深工程師視角的韌性設計（重點）

這份作業特別以「應對各種異常狀況」為目標。**核心原則：MySQL 是唯一的硬性依賴，Redis 與 RocketMQ 都是「盡力而為」的軟性依賴。**

### 失敗情境對照表

| 情境 | 一般寫法的問題 | 本專案的處理 |
|---|---|---|
| **Redis 掛掉**時查詢 | 直接噴 500 | 視為快取未命中 → 改讀 MySQL → 正常回 200 |
| **Redis 掛掉**時寫入 | 噴 500 | 快取操作變 no-op，DB 照常寫入，回成功 |
| **RocketMQ broker 掛掉**（啟動時） | 整個 App 起不來 | 啟動失敗只記 log，App 照常啟動 |
| **RocketMQ broker 掛掉**（送訊息時） | 請求卡住 / 500 | 送訊息有 3 秒逾時上限，失敗記 log，建立仍回 201 |
| **MySQL 掛掉** | 500（正確） | 500 — DB 是真相來源，屬硬性依賴，乾淨地拋錯 |
| **更新後 recent 快取沒更新** | `/recent` 顯示舊主旨 | 更新時讓 recent 失效，下次讀取由 DB 重建 |
| **刪除後 recent 還留著舊資料** | 已刪的還出現在 `/recent` | 刪除時讓 recent 失效，下次讀取由 DB 重建 |
| **並發寫入導致 recent 超過 10 筆** | 無限增長 | 每次 LPUSH 後 LTRIM 0..9 |
| **type 錯誤 / 欄位過長 / 收件人格式錯** | DB 截斷錯誤變 500，或存進髒資料 | 提早擋下回 400（Bean 驗證 + 列舉解析 + 依類型檢查收件人格式） |
| **查不到的 ID** | 不一致 / 500 | 統一回 404 |
| **JSON 格式壞掉** | 500 | 對應到 400 |
| **未預期的例外** | 噴 stack trace 給前端 | 統一回 JSON 格式的 500，不洩漏堆疊 |
| **SQL injection** | 有風險 | 全部用 JdbcTemplate 參數化查詢 |

### 一致性模型（設計取捨）
- MySQL 是系統真相來源（system of record）。
- Redis 是旁路快取（cache-aside），允許「最終一致」，由 TTL + 明確失效控制。
- RocketMQ 是 fire-and-forget，至多一次投遞。
- 若要「保證投遞」，正解是 **transactional outbox 模式**（同一交易寫入 outbox 表再非同步轉送）— 本作業範圍外，但面試可提。

## 五、怎麼驗證它能跑（沒有 UI，但可手動驗）

這是後端 API，沒有畫面，但可以用 `curl`、Postman，或 GET 端點直接用瀏覽器打開來驗。

### 啟動
```bash
docker compose up -d        # 起 MySQL / Redis / RocketMQ
./mvnw test                 # 跑 23 個測試（不需要 infra 也能跑）
./mvnw spring-boot:run      # 啟動 App（需要 Java 21）
```

### 手動測試（curl）
```bash
# 1. 建立（預期 201）
curl -i -XPOST localhost:8080/notifications -H 'Content-Type: application/json' \
  -d '{"type":"email","recipient":"user@example.com","subject":"Welcome!","content":"Thanks!"}'

# 2. 依 ID 查（預期 200；不存在 404）
curl localhost:8080/notifications/1

# 3. 最近清單（瀏覽器也能開）
curl localhost:8080/notifications/recent

# 4. 更新（預期 200；不存在 404）
curl -i -XPUT localhost:8080/notifications/1 -H 'Content-Type: application/json' \
  -d '{"subject":"Updated","content":"Updated body"}'

# 5. 刪除（預期 204；不存在 404）
curl -i -XDELETE localhost:8080/notifications/1

# 錯誤情境（都應回 400）
curl -i -XPOST localhost:8080/notifications -H 'Content-Type: application/json' \
  -d '{"type":"fax","recipient":"user@example.com"}'            # type 錯誤
```

### 驗證三個元件都有作用
- **MySQL**：`docker exec mysql mysql -utaskuser -ptaskpass taskdb -e "SELECT * FROM notification;"`
- **Redis**：`docker exec redis redis-cli keys 'notification*'`
- **RocketMQ**：開瀏覽器 http://localhost:8088 → 看 topic `notification-topic` 有沒有訊息

### 自動化測試覆蓋
共 **23 個測試全通過**：
- `NotificationTypeTest`（2）：類型解析與拒絕未知值
- `NotificationServiceTest`（11）：核心邏輯、快取旁路、更新/刪除失效、收件人驗證
- `NotificationControllerTest`（9）：5 個端點的狀態碼（201/200/204/400/404）
- `DemoApplicationTests`（1）：在 MySQL/Redis/MQ 全部關閉下仍能載入 context（證明韌性設計有效）
